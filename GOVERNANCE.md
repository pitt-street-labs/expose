# EXPOSE Governance

This document describes how the EXPOSE project is governed, how decisions are made, and how contributions are structured.

## Project leadership

EXPOSE is developed and maintained by **Pitt Street Labs** (a Korlogos project). The project lead makes final decisions on architecture, roadmap, and releases.

## Decision-making

### Architecture Decision Records (ADRs)

Significant architectural decisions are documented as ADRs in `docs/adr/`. Each ADR captures the context, decision, consequences, and alternatives considered. ADRs are the authoritative record for design decisions -- they are locked once accepted and are not modified without a dedicated review session.

Current ADRs cover: implementation language (ADR-001), graph storage (ADR-002), deployment posture (ADR-003), output artifact format (ADR-004), LLM integration boundaries (ADR-005), repository and licensing (ADR-006), multi-tenancy (ADR-007), authorized use and ethics (ADR-008), commercial structure (ADR-009), and FedRAMP-ready posture (ADR-010).

New ADRs may be proposed via pull request. The project lead accepts or rejects ADR proposals.

### Issues and feature requests

Day-to-day decisions -- bug fixes, minor improvements, new collectors, documentation updates -- are tracked through GitHub Issues. Feature requests should include a clear use case description. The project lead and maintainers assess scope fit before approving implementation work.

## Contribution tiers

### Community contributor

Anyone who submits a pull request, files an issue, or participates in discussions. Community contributors follow the process described in [CONTRIBUTING.md](CONTRIBUTING.md), including DCO sign-off on all commits.

### Maintainer

Maintainers have write access to the repository and review pull requests. Maintainers are appointed by the project lead based on sustained, high-quality contributions and alignment with the project's goals and ethics. Maintainer responsibilities include:

- Reviewing and merging pull requests within the documented SLA (5-10 business days for initial response).
- Triaging issues and applying the label taxonomy (`epic:`, `area:`, `priority:`, `type:`).
- Upholding the Code of Conduct and Ethics guidelines.
- Ensuring that changes respect the specification, ADRs, and architectural boundaries.

Maintainer status may be revoked if a maintainer becomes inactive or violates project standards.

## Release process

Releases follow [Semantic Versioning](https://semver.org/):

- **Major** (e.g., 1.0.0 to 2.0.0) -- breaking changes to schemas, configuration format, or API contracts.
- **Minor** (e.g., 1.0.0 to 1.1.0) -- backward-compatible additions such as new collectors, new attribution rule predicates, or new API endpoints.
- **Patch** (e.g., 1.0.0 to 1.0.1) -- bug fixes, security fixes, and documentation corrections with no behavior changes.

Pre-release tags use `-rc.N` and `-beta.N` suffixes.

### Artifact signing

All release artifacts -- container images, SBOMs, and release binaries -- are signed with [cosign](https://docs.sigstore.dev/cosign/overview/) using keyless signing via Sigstore. SLSA provenance attestations are included. Verification instructions are in [SECURITY.md](SECURITY.md).

### SBOM and supply chain

Every container image ships with an SPDX-format Software Bill of Materials generated by [syft](https://github.com/anchore/syft). Build provenance targets SLSA Level 2, with Level 3 as an ongoing improvement goal.

## Open-core structure

EXPOSE follows an open-core model with four product surfaces, as defined in ADR-009:

| Product surface | License | Repository |
|----------------|---------|------------|
| **EXPOSE Core** | Apache 2.0 | Public -- the engine, collector framework, builtin collectors, attribution rules, Helm chart, schemas, and documentation |
| **EXPOSE Threat Context** | Proprietary (Korlogos commercial EULA) | Private -- APT targeting profiles, dark-web correlation, historical enrichment, adversary infrastructure detection |
| **EXPOSE Identity Surface** | Proprietary (Korlogos commercial EULA) | Private -- WHOIS-personnel correlation, authorized social-media reconnaissance, personnel-graph attribution (higher ethics bar, separate licensing) |
| **EXPOSE Research** | CC BY 4.0 (data) / Apache 2.0 (tooling) | Public -- reference graph datasets, benchmark datasets, reference rule packs |

The Core repository has no dependencies on proprietary code. Commercial modules consume Core's published artifacts through the documented integration surface (the Apache 2.0 SDK). This ensures Core can be used independently and that commercial modules do not create obligations under the Apache 2.0 license.

Community contributions go to EXPOSE Core. The commercial modules are developed by Pitt Street Labs and are not open for external contributions. Revenue from commercial modules funds continued development of EXPOSE Core.

## Ethics oversight

The project maintains an [ETHICS.md](ETHICS.md) document that defines intended use, non-goals, and explicit prohibitions. Ethics review is triggered by changes that affect what data is collected, how authorization scope is enforced, or how attribution decisions are made. A quarterly review cycle ensures the ethics framework remains current as the project evolves.

## Amendments

This governance document may be updated by the project lead. Significant changes (such as altering the contribution tier structure or the open-core boundary) will be communicated through a dedicated issue or announcement before taking effect.
